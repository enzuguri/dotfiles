# Agent Harness

A portable Claude Code configuration. `install.sh` symlinks the contents of this directory into `~/.claude/`, so the same harness travels with the dotfiles repo across machines and is version-controlled like any other config.

This README is the entry point for colleagues. If you only want to know how to install it, jump to [Install](#install). If you want to understand why it is shaped this way, read the rest.

---

## Relation to the dotagents standard

The directory layout follows [bgreenwell/dotagents](https://github.com/bgreenwell/dotagents) — a draft community standard for organising agent context under a hidden `.agents/` directory at the repo root. The high-level idea matches ours: **`.agents/` is a store for information generated by, or consumed by, agents** — kept out of the project root, organised by lifecycle rather than by tool vendor.

Where the standard prescribes specific subfolders (`rules/`, `context/`, `logs/`, `memory/`, `skills/`, `specs/`, `personas/`), we adopt the ones that fit our pipeline (`rules/`, `context/`, `logs/`, `skills/`) and add an `agents/` subfolder for sub-agent prompt definitions. `dotagents` is Draft 0.1.0 and explicitly invites adaptation, so this is a conformant extension rather than a divergence.

---

## The core idea

**Context is the scarce resource, not tokens.**

Frontier LLMs lose coherence past roughly 150–200 instructions or once context fills with low-signal material. Larger context windows do not fix this — they delay the failure mode. A 1M-token window that is 70% full of tool output, half-read files, and prior reasoning steps performs worse than a 200k window at 30% utilisation.

Every design choice in this harness follows from that observation:

- **Hard constraints first.** `AGENTS.md` opens with non-negotiable rules. Constraints buried late get silently skipped by the model — the order of the prompt encodes priority.
- **Sub-agents are context firewalls, not personas.** A sub-agent exists to run a bounded job in its own context window and return a compact summary. Its purpose is to keep the orchestrator's context clean, not to roleplay a specialism.
- **Artifacts over conversation.** Agents coordinate through files under `.agents/` (and a few in `~/.config/`), not by passing prose through the orchestrator. The orchestrator reads the artifact path; the artifact itself never enters the orchestrator's context unless needed.
- **Compact targets.** Aim for under 40% context utilisation, persist progress at 60%, resume from the summary — never from the transcript.

## The pipeline

Non-trivial work flows through a fixed sequence of stages. Each stage produces one artifact under `.agents/logs/<task-slug>/`; the next stage reads it directly.

```
research-agent ──► design-discussion ──► Plan ──► implementation ──► verification-agent ──► git-agent
   logs/<slug>/         logs/<slug>/         (in-conv.)                  (structured report)        (PR / commit)
   research.md          constraints.md
```

The artifact convention matters because it inverts the usual flow. Instead of each stage handing the next a prose summary that the orchestrator must hold, each stage drops a structured file in `.agents/logs/<task-slug>/`. Downstream stages read what they need from that file. The orchestrator only needs to remember the slug.

### Stages

| Stage | Role | Deliverable |
|---|---|---|
| `research-agent` | Parallel information gathering. Set aside the goal; document what the code *is*. Surface inconvenient facts. | `.agents/logs/<slug>/research.md` with one section per source |
| `explore-agent` | Read-only codebase orientation. Grep before read; 8-file Read budget; structured output schema. | Structured summary (entry points, modules, conventions, boundaries, hotspots, gotchas) — persisted to `.agents/logs/<slug>/exploration.md` for non-trivial runs |
| `design-discussion` | Takes research + goal, produces architectural constraints *before* a plan exists. The "brain surgery" stage. | `.agents/logs/<slug>/constraints.md` with `Locked-in / Flexible / Acceptance criteria` |
| `Plan` (built-in) | Step-by-step implementation plan that satisfies the constraints document. | In-conversation plan |
| `verification-agent` | Lint / format / typecheck / test / build in parallel. Reads commands from `.agents/context/project-tools.md`. | Structured pass/fail report with verdict |
| `git-agent` | Commits, branches, rebases, PRs, CI monitoring. Never reads source — works from the prompt summary and `git log` only. | Commit / PR / status |
| `slack-insights` | Peer-interest mining, channel digests, period rollups. Out-of-band but uses the same artifact pattern. | Daily/period reports under `~/.config/slack-insights/reports/` |

### Why this shape

- **Research before goal.** If you fit the codebase to the goal during research, you lose the chance to discover that the goal is wrong for the codebase. `research-agent` documents what *is*; `design-discussion` is the first stage that gets to look at the goal.
- **Constraints before plan.** A well-written plan feels validating but only verifies that the plan is well-written — not that it fits the codebase. The constraints document is what the plan must satisfy; it makes plan review checkable rather than vibes-based.
- **Verification is non-negotiable.** `quick` mode after every edit, `full` mode before declaring complete. Verification reads commands from `.agents/context/project-tools.md` so checks match what CI actually runs — no inference from `package.json`, no guessing.
- **Plan approval ≠ code approval.** The harness states this explicitly because models (and humans) conflate the two. Every diff must be read.

## Layout

```
.agents/
├── AGENTS.md          → symlinked to ~/.claude/CLAUDE.md (top-level instructions)
├── agents/            → symlinked to ~/.claude/agents/   (sub-agent definitions)
│   ├── research-agent.md
│   ├── explore-agent.md
│   ├── design-discussion.md
│   ├── verification-agent.md
│   ├── git-agent.md
│   ├── slack-insights.md
│   └── bourgeoisie-reviewer.md
├── rules/             → symlinked to ~/.claude/rules/    (behavioural guidance fragments)
│   ├── code-style.md
│   ├── types.md
│   ├── boundaries.md
│   ├── project-conventions.md
│   ├── error-handling.md
│   ├── tooling.md
│   └── ast-grep.md
├── skills/            → symlinked to ~/.claude/skills/   (invocable harness skills)
│   └── discover-project-tools/
└── install.sh
```

At runtime, two more subdirectories appear in each consumer repo:

```
<consumer-repo>/.agents/
├── context/           # Generated reference data, regenerated wholesale
│   ├── project-tools.md       # Written by /discover-project-tools, read by verification-agent
│   └── boundaries.md          # Cached output of the abstraction-boundary discovery algorithm
└── logs/              # Per-task handoff artifacts
    └── <YYYY-MM-DD>-<task-slug>/
        ├── research.md
        ├── constraints.md
        └── exploration.md
```

### Five kinds of content, deliberately separated

- **Agents** (`agents/`) — context firewalls with their own prompts, tool allowlists, and protocols. Invoked via the Agent tool. Shipped with the harness.
- **Rules** (`rules/`) — shared behavioural guidance loaded on demand by topic (e.g. `code-style` before any Write/Edit, `boundaries` when designing new modules). Summaries live in `AGENTS.md` so the orchestrator knows when each becomes relevant; full text is read only when the topic is in play. Not invocable as skills — they are prompt fragments. Shipped with the harness.
- **Skills** (`skills/`) — actual invocable harness skills. Currently `discover-project-tools`, which writes `.agents/context/project-tools.md`. Shipped with the harness.
- **Context** (`context/`) — generated reference data, per-repo, regenerated wholesale when stale. Useful to teammates because it encodes verification commands and codebase structure; commit decision is per-consumer-repo. `project-tools.md` and `boundaries.md` live here. Not shipped — produced at runtime.
- **Logs** (`logs/`) — per-task handoff artifacts that pipeline stages drop for the next stage. One subdirectory per task slug. Typically gitignored — these are working memory for one task, not durable team context. Not shipped — produced at runtime.

The distinction matters: if you treat rules fragments as skills, you get noisy skill lists and accidental invocations. If you bake rule text directly into each agent prompt, you duplicate guidance and drift between copies. If you commingle generated context with per-task scratch, you cannot decide what to commit.

## `.agents/` — what lives there, why it lives there

The mental model:

| Subdir | Lifecycle | Who writes | Who reads | Typical fate |
|---|---|---|---|---|
| `agents/` `rules/` `skills/` | Hand-authored, evolves slowly | Humans | Agents | Committed (shipped with the harness) |
| `context/` | Generated, regenerated wholesale on staleness | Tooling (skills, discovery algorithms) | Agents | Commit decision per repo — useful to teammates, but auto-regenerable |
| `logs/` | Generated per task, lives for the duration of that task | Pipeline-stage agents | Next pipeline stage; orchestrator on resume | Usually gitignored — working memory |

The `context/` vs `logs/` split is the key one to internalise:

- `context/` is **snapshot reference data**. If you delete it, a skill or algorithm rebuilds it with no information loss. Similar in spirit to `dotagents`' `context/` slot for static schema/API data — ours is generated rather than hand-authored, but the consumer role is identical: read-only reference.
- `logs/` is **per-task handoff scratch**. If you delete it mid-task, you lose pipeline-stage outputs and the next agent has to start over.

If you ever add per-project ADRs or learned user preferences, those would belong in `.agents/memory/` per the dotagents spec — but we don't have any yet, so the directory isn't created.

## Conventions worth calling out

- **Reply concisely.** The communication-style block in `AGENTS.md` assumes a principal-engineer audience: skip basics, focus on trade-offs, explain *why* for architectural choices.
- **Never destructive without confirmation.** `reset --hard`, `push --force`, `branch -D`, `clean -fd` require explicit user confirmation. `git-agent` knows this.
- **No magic-word triggers.** Agents do not branch behaviour on specific phrases in the prompt. The protocol applies to any well-formed call. Triggers are brittle and surprise users.
- **Hard rules listed first.** Inside every agent prompt, non-negotiable rules come before procedure. This is the same reason `AGENTS.md` opens with constraints — late instructions get skipped.

## Install

```bash
./install.sh
```

The script symlinks `AGENTS.md`, `agents/`, `rules/`, and `skills/` into `~/.claude/`. Existing files at those paths are backed up with a timestamp; existing symlinks pointing to the same target are left alone.

Reverse with `rm ~/.claude/{CLAUDE.md,agents,rules,skills}` — only symlinks are touched, and any pre-install files survived as `*.bak.<timestamp>`.

Note: `context/` and `logs/` are *not* symlinked. They appear inside each consumer repo's own `.agents/` directory at runtime.

### Gitignore guidance for consumer repos

The harness doesn't enforce a policy on what to commit. Per-task `logs/` are working memory and rarely useful to teammates; `context/` is auto-regenerable but encodes verification commands and codebase structure, which can be helpful to share. A reasonable default for most repos:

```gitignore
# Agent harness — per-task scratch
.agents/logs/

# Optional: also ignore generated context if you'd rather regenerate per-clone
# .agents/context/
```

Commit `.agents/context/` if you want teammates (and CI) to pick up the same verification commands and boundary discoveries without re-running `/discover-project-tools` on every clone. Ignore it if you'd rather treat it strictly as a per-developer cache.

## When to deviate

The harness is opinionated, but the opinions are observable in `AGENTS.md`. If a convention does not fit your repo:

- **Edit `AGENTS.md` for global changes** — it is the top-level prompt for every conversation.
- **Edit `agents/<name>.md` for stage-specific changes** — each agent is self-contained; protocols can be tightened or relaxed without affecting the others.
- **Add rules fragments under `rules/`** for shared guidance that multiple agents need to load — but resist if only one agent needs it; inline it there instead.

The cost of a wrong convention is real: agents apply it silently across every task. Treat changes to `AGENTS.md` and the agent prompts the same way you would treat changes to a CI config — small, reviewed, and reversible.
